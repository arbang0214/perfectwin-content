## 1️⃣ GSC 데이터 깊이 보강

### 검색어 × 페이지 페어 수집 추가
GSC API의 `dimensions: ["query", "page"]` 두 차원 동시 호출로 "어떤 키워드로 검색해서 어떤 포스트가 클릭됐는지" 직접 매핑.

| 이전 | 이후 |
|---|---|
| 검색어 Top N · 페이지 Top N 별도 수집 (매칭 불가) | 검색어×페이지 페어 row 수집 (직접 매칭) |

### 모든 GSC 표에 자동 해석 컬럼
position·clicks·impressions 룰 기반으로 한 줄 해석 자동 생성. 예시:

- `첫 페이지. 5번 중 1번 클릭 (CTR 20%) — 의미 있는 트래픽`
- `첫 페이지. 18번 노출, 클릭 0 — 메타·제목 최적화 기회`
- `둘째 페이지. 12번 노출, 클릭 0 — 순위 올리면 클릭 시작`
- `셋째 페이지 이하. 26번 노출, 클릭 0 — 사실상 미노출`

홈페이지 검색어 Top 5 / 블로그 인기 컨텐츠 Top 5 / 새 검색어→포스트 매칭 표 모두 적용. 토큰 없이 결정적으로 생성.

## 2️⃣ 측정 출처·시점 차이 명시

### 배경
ARUM이 5/18 리포트에서 "GSC 클릭 0인데 Organic 5건은 어디서?" 의문 제기. 두 데이터 소스가 측정 정의·시점이 달라 어긋났던 케이스.

- **inblog**: 블로그 플랫폼에서 referrer가 검색엔진인 세션 카운트, **당일 실시간**.
- **GSC**: 구글이 알려주는 검색결과 클릭, **2~3일 지연** (콜렉터가 자동 fallback).

### 적용한 라벨링
**표 헤더 옆 출처·기준일 명시:**
- GSC 표: `_(2026-05-16 기준 · GSC 처리 지연으로 2일 fallback)_`
- inblog 표: `_(inblog 실시간 측정)_`

**블로그 섹션 도입부에 매번 안내문 출력:**
> "몇 건 들어왔는지" (방문·CTA 클릭·Organic 비중) → inblog 실시간. **당일 데이터**.
> "어떤 검색어로 어떤 포스트에 들어왔는지" (GSC 표·검색어→포스트 매칭) → GSC 처리 지연으로 **2~3일 전 데이터**.

**Organic 행 의미 컬럼에 정의 차이 주석:**
> referrer가 검색엔진(google.com 등)인 세션. GSC '검색결과 클릭'과 정의가 달라 일치하지 않을 수 있음 (Discover·News·직접 링크 포함).
